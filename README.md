## 작업 위임 및 실행 방법
이 워크스페이스는 단일 에이전트 호출부터 18인 에이전트 군단의 풀 오토메이션까지, 프로젝트의 규모와 요구되는 통제 수준에 따라 5가지의 유연한 작업 위임 방식을 지원합니다.

### 1. Top-Down Orchestration (오케스트레이터 전권 위임)
프로젝트의 시작부터 끝까지 개입 없이 자동화하는 가장 강력한 방식입니다. 요구사항이 명확하고, 빠르고 독립적인 프로토타이핑이 필요할 때 사용합니다. Orchestrator가 전체 8단계(8-Phase) 파이프라인에 따라 하위 에이전트들을 지능적으로 병렬 호출합니다.

```bash
# 오케스트레이터에게 전체 프로젝트 생성 지시
gemini -y -p "/agents:orchestrator 사용자 인증 기능이 포함된 대시보드 웹 앱을 기획부터 구현까지 완료해줘."
```

### 2. Phased Gate Control (단계별 인간 개입 및 제어)
핵심 마일스톤마다 작업자(Human)가 결과를 검수(QA)하고 방향을 미세 조정한 뒤, 다음 단계의 에이전트를 직접 호출하는 방식입니다. 품질 관리가 매우 중요하거나 기획 단계에서 잦은 변경이 예상될 때 적합합니다.

```bash
# 1단계: PM 에이전트를 호출하여 초안 기획 (이후 작업자가 spec.md 리뷰 및 수정)
gemini -y -p "/agents:pm 사내 도서 관리 시스템에 대한 요구사항 명세서 작성"

# 2단계: 확정된 기획을 바탕으로 Architect에게 시스템 설계 지시
gemini -y -p "/agents:architect 확정된 spec.md를 바탕으로 API 및 DB 스키마 설계"

# 3단계: Task Planner를 통해 작업 분해 후 Task Runner로 병렬 구현 시작
gemini -y -p "/agents:task-planner 설계 문서를 바탕으로 개발 태스크 생성"
gemini -y -p "/agents:task-runner 대기 중인 모든 태스크 병렬 실행"
```

### 3. Direct Micro-Tasking (도메인별 마이크로 태스킹)
거대한 프로젝트가 아닌, 특정 도메인의 단일 업무만 개별 전문 에이전트에게 맡기는 방식입니다. 에이전트를 마치 마이크로서비스 도구처럼 활용하여 생산성을 극대화합니다.

```bash
# 버그 추적 및 디버깅 전담
gemini -y -p "/agents:debugger src/server/auth.js 파일에서 발생하는 메모리 누수 원인 분석 및 해결책 적용"

# 기술 문서 및 보고서 생성 전담
gemini -y -p "/agents:docs 최근 완료된 태스크 로그를 바탕으로 경영진 보고용 docs/report.pptx 생성"
```

### 4. Interactive Co-authoring (대화형 브레인스토밍)
자동 승인 플래그(-y)를 제외하고, 기획자(PM)나 리서처(Scout) 에이전트와 대화하며 아이디어를 구체화하는 방식입니다. 요구사항이 모호할 때 에이전트의 주도적인 질문을 통해 문서의 빈틈을 채우고 고품질의 산출물을 도출합니다.

```bash
# -y 플래그 없이 실행하여 대화형 모드 진입
gemini -p "/agents:pm 새로운 하이엔드 B2B SaaS 솔루션 기획을 도와줘"
```

### 5. Task Queue Injection (JSON 태스크 수동 주입)
에이전트의 자동 기획을 거치지 않고, 작업자가 직접 .gemini/agents/tasks/ 디렉토리에 task-*.json 형태의 작업 지시서를 작성하여 큐(Queue)에 넣은 뒤 실행 엔진만 가동하는 방식입니다. CI/CD 파이프라인 연동이나 정형화된 반복 스크립트 작업에 유용합니다.

```bash
# 1. 사용자가 직접 task-custom-01.json 생성 (status: "pending")
# 2. Task Runner 엔진 가동하여 큐에 있는 작업만 정확히 실행
gemini -y -p "/agents:task-runner 펜딩된 작업 큐 순차적/병렬적 실행"
```

## 100% 활용을 위한 가이드 (Best Practices)
17명의 전문 에이전트가 유기적으로 협업하는 이 시스템을 가장 안전하고 효율적으로 사용하기 위한 핵심 지침입니다.

### 초기 컨텍스트 주입에 시간을 투자하세요

Orchestrator나 PM에게 첫 업무를 지시할 때, "게시판 만들어줘"와 같은 단답형 지시를 피하세요. 비즈니스 목적, 타겟 유저, 핵심 기능, 참고할 벤치마크 URL 등을 텍스트로 충분히 제공(Dump)해야 합니다. 첫 지시서의 디테일이 18명 전체 에이전트의 최종 산출물 퀄리티를 결정합니다.

### 풀 오토메이션보다 '단계별 승인(Phased Approval)'을 권장합니다

처음부터 끝까지 한 번에 실행하면 환각(Hallucination)으로 인한 오류가 누적될 수 있습니다.

기획(PM)과 설계(Architect, Designer) 단계까지 먼저 실행하여 산출물(.md)을 생성한 뒤, 작업자(Human)가 이를 검토 및 미세 조정하고 Task Runner를 통해 실제 코드 구현을 시작하는 것이 가장 안전합니다.

#### 오케스트레이터 범위 제한 지시 (프롬프트 제어 방식)
오케스트레이터의 강력한 자동화를 활용하되, **프롬프트에 명확한 정지 조건(Boundary)**을 부여하는 방식입니다.


```Bash
gemini -y -p "/agents:orchestrator [프로젝트 상세 설명]. 

단, 너의 워크플로우 8단계 중 Phase 1(Recon(사전탐색) & Plan(계획))과 Phase 2(Architecture(설계) & Design(디자인))까지만 완료해. 
절대 하위 에이전트(backend, frontend 등)를 호출하여 실제 코드를 구현하는 Phase 3 단계로 넘어가지 말고 작업을 종료한 뒤 나에게 보고해."
```

이 방식의 특징:

오케스트레이터가 알아서 scout, pm, architect, designer를 적절히 호출하여 기획/설계 문서를 세팅합니다.

지시한 대로 구현 단계 직전에 멈추기 때문에, 작업자가 결과물을 확인하고 수정할 타이밍을 벌 수 있습니다.

검토가 끝난 후, 다시 **오케스트레이터**나 **task-runner**를 호출하여 "설계가 승인되었으니 다음 단계(Phase 3)부터 이어서 진행해"라고 지시하면 됩니다.

### 관제탑(Status Reporter)을 적극 활용하세요

수시로 gemini -y -p "/agents:status 현재 전체 공정률과 병목 구간 보고해줘" 명령어를 통해 작업 현황을 대시보드 형태로 모니터링하세요. 어떤 에이전트 단계에서 주로 실패([failed])가 발생하는지 파악하여 문제를 조기 차단할 수 있습니다.

### 컨텍스트 오염을 주기적으로 관리하세요

에이전트가 엉뚱한 코드를 작성하거나 과거의 실수를 반복한다면, .gemini/memory/global.md나 src/docs/context.md 파일이 너무 방대해졌거나 오염되었을 확률이 높습니다. 작업자가 개입하여 이미 완료된 오래된 맥락은 지우고 핵심 정보(SSOT)만 간결하게 남겨주면 에이전트의 지능이 크게 향상됩니다.

### 토큰 가성비를 최적화하세요

프로젝트의 주요 마일스톤이 끝날 때마다 .gemini/logs/delegation-ledger.md의 토큰 사용량을 분석하세요. 특정 에이전트가 과도한 토큰을 소모하고 있다면, 해당 에이전트의 .toml 프롬프트나 참조하는 SKILL.md의 길이를 줄여 비용 효율을 높여야 합니다.


## 부분개발 시 유의사항
1. 기존 백엔드 기반 프론트엔드만 작업할 때 (부분 개발)
    - 이미 완성된 API나 백엔드 코드가 있다면, 프론트엔드 파트만 독립적으로 가동할 수 있습니다.
    - 사전 준비: 기존 백엔드의 API 명세서를 src/docs/api-spec.md에 넣어두거나, 글로벌 메모리(.gemini/memory/global.md)에 현재 백엔드 상태를 상세히 적어둡니다.
    - 실행 방식: 오케스트레이터를 부르지 않고, Designer와 Frontend 에이전트만 직접 호출합니다.

```Bash
# 1. 디자이너에게 기존 백엔드 컨셉에 맞는 UI 테마 설계 지시
gemini -y -p "/agents:designer 기존 백엔드 API 데이터를 보여줄 대시보드의 시각적 테마를 설계해줘"

# 2. 프론트엔드 에이전트에게 API 명세서를 연동한 UI 구현 지시
gemini -y -p "/agents:frontend src/docs/api-spec.md를 기반으로 React 대시보드 UI를 구현하고 API를 연동해"
```

2. 기존 소스코드 유지보수 및 리팩토링 할 때

    - 기존에 짜인 레거시(Legacy) 코드를 분석하고 개선하거나 버그를 잡을 때도 에이전트들은 매우 강력합니다.

    - Scout (정찰병) 활용: scout 에이전트는 프롬프트(.toml) 자체에 [Legacy Intelligence] 임무가 부여되어 있습니다. 코드를 만지기 전에 먼저 파견하여 기존 코드의 의도와 의존성을 파악하게 하세요.


```Bash
gemini -y -p "/agents:scout 현재 src/client 폴더 내의 레거시 코드 구조를 분석하고, 컴포넌트 분리를 위한 리팩토링 리포트를 작성해줘"
```

    - Debugger & Optimizer 투입: 정찰이 끝나면, 버그 수정이나 성능 개선을 전문 에이전트에게 맡깁니다.

```Bash
gemini -y -p "/agents:optimizer src/server/api.js 파일의 DB 조회 쿼리에서 발생하는 N+1 문제를 분석하고 최적화해줘"
```

### 유지보수 시 핵심 주의사항 (Scope Boundaries)
유지보수 작업을 지시할 때는 에이전트가 멀쩡한 코드를 건드리지 않도록 작업 범위(Scope)를 엄격하게 제한하는 것이 가장 중요합니다.

- 나쁜 지시: "로그인 버그 좀 고쳐줘" (에이전트가 전체 코드를 뜯어고칠 위험이 있음)
- 좋은 지시: "현재 src/components/Login.jsx 파일에서 발생하는 상태 관리 버그를 찾아 수정해. 단, 다른 파일이나 폴더는 절대 수정하지 마."
